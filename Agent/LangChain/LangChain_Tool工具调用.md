# Tool 工具调用

## 先给定位

Tool 是把外部能力暴露给模型调用的接口。模型本身只会生成文本，工具让它可以搜索、查数据库、调用业务 API、运行计算或读取文件

类比人类工作，模型负责判断和表达，工具负责接触现实世界

## Tool 的核心

一个工具至少要说明三件事：

- 名称：模型用来识别工具
- 描述：模型判断什么时候该用它
- 参数 schema：模型按什么结构传参

如果工具描述含糊，模型可能会误用工具。如果参数 schema 太松，后端就要承担更多校验成本

## Tool 和函数的区别

普通函数是给程序员调用的，Tool 是给模型选择和调用的。它不仅要能运行，还要能被模型理解

所以工具设计要像 API 设计一样严谨：名字明确、输入窄、输出稳定、错误可解释、权限可控制

## 常见风险

- 工具权限过大，模型一次误判就能触发真实副作用
- 工具结果直接信任，导致数据污染或提示注入
- 工具太多，模型选择成本变高
- 工具描述不清，调用频率和参数质量不稳定

更稳的做法是把高风险操作拆成人工确认、只读工具、写入工具和审计日志
